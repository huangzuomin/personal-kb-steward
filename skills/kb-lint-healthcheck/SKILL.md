# Skill：kb-lint-healthcheck

## 定位

知识库健康检查。它是治理 Skill，不是内容生产 Skill。

目标是发现知识库腐烂、链接失效、来源缺失、结构失控和质量下降的风险。

## 触发

- 用户要求检查知识库健康
- 每周例行检查
- 大批量生成或整理之后
- 用户怀疑存在断链、重复、混乱、过期内容

## 输入

扫描范围：

- `quicknote/`
- `inbox/`
- `raw/`
- `wiki/`
- `outputs/`

跳过：

- `.obsidian/`
- 明确归档且无需检查的目录

## 输出

位置：

```text
outputs/
```

页面类型：

```text
lint-report
```

文件命名：

```text
kb-lint-YYYY-MM-DD-HHMMSS.md
```

## Lint Report 元数据

```yaml
---
title:
type: lint-report
status: compiled
stage: compiled
created:
updated:
sources:
related:
tags:
risk_level:
review_required:
---
```

`status` 只能使用：

```text
compiled
manual_review
archived
```

## 检查项

- 断链
- 重复标题
- 重复来源
- `sources` 为空
- `sources` 过粗，例如只写 `raw/`
- 生成页缺少 YAML 元数据
- 状态值不合法
- `quicknote/`、`inbox/` 积压
- 长期未更新页面
- 占位栏目未完成
- 没有入链的孤立页面
- 主题页来源不足
- 材料包缺少事实、案例、反方、时间线、缺口

## 风险分级

```text
P0 破坏性风险：原始资料可能被覆盖、删除或不可追溯
P1 可信度风险：无来源结论、断链、来源不存在
P2 结构风险：重复页、错误分类、状态异常
P3 维护风险：积压、孤立页、过期、占位内容
```

## 正文结构

```markdown
# 知识库健康检查报告

## 总览

## P0 风险

## P1 风险

## P2 风险

## P3 风险

## 可自动修复建议

## 需要人工确认

## 下次检查建议
```

## 方法

1. 建立全库文件索引。
2. 解析所有 YAML 元数据。
3. 解析所有 `[[wikilink]]` 并确认能否落到真实文件。
4. 检查 `sources` 是否存在、是否具体到文件。
5. 检查状态值、页面类型、必要栏目和占位内容。
6. 按 P0/P1/P2/P3 给问题分级。
7. 输出可自动修复建议和必须人工确认的问题。

## 自动修复边界

允许：

- 生成报告
- 标记派生页为 `manual_review`
- 为派生页补充确定的缺失元数据
- 给出修复建议

禁止：

- 删除原始资料
- 移动原始资料
- 自动合并主题页
- 自动重写来源内容
- 自动删除重复页

## 质量标准

- 每个问题必须有文件路径。
- 每个高风险问题必须有原因和建议动作。
- 不能只输出统计数字。
- 不确定的问题必须进入“需要人工确认”。

## 失败处理

- 文件无法读取：记录路径和错误，不中断整个检查。
- 元数据无法解析：标记为 P1 或 P2。
- 检查范围过大：先输出抽样报告并建议分批检查。

## 禁止事项

- 不删除原始资料。
- 不移动原始资料。
- 不自动合并页面。
- 不自动改写 raw、quicknote、inbox。
- 不把健康检查报告伪装成内容整理成果。
